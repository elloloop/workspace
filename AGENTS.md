# Engineering rules for this repository

These rules apply to **every** change — agent-driven or human. They are
not aspirations. Code that violates them is rejected, not landed and
"cleaned up later."

## 1. No patch fixes. No shims. No compatibility layers.

If the existing shape is wrong, change it. Do not wrap it. Do not add a
"thin translation layer" so the wrong shape can keep being called.
Examples of what is forbidden:

- Keeping a `Client` / `Adapter` / `Wrapper` interface whose only purpose
  is to mirror the previous API surface so callers don't have to be
  rewritten.
- Witness tables, typeID lookups, dynamic dispatch — when the call site
  knows the concrete type, use the concrete type.
- `map[string]any` payloads bridging typed code to typed code.
- "Old caller still works because of this fallback" comments.

If a refactor exposes 200 call sites that need updating, update them.
That is the job. Saving an afternoon by leaving a shim costs everyone
who reads the code afterwards.

## 2. Delete dead code. Always.

When a refactor lands, remove:

- Unused functions, methods, types, constants, variables.
- Old files made obsolete by the new structure.
- Compatibility re-exports kept "in case someone is still importing
  the old path."
- Comments referencing the previous design ("used to be X, now Y").
- TODOs whose author left the project — either do them or delete them.

`grep` for the symbol before claiming it's gone. If a deleted file is
still referenced by anything, that's a real link to fix, not a reason
to keep the file.

## 3. Refactor at the right level.

When you change a function signature, propagate the change to every
call site. Do not introduce a parallel "v2" function while the old one
lingers. Do not keep an `isLegacy bool` parameter to switch between
behaviours.

The exceptions are real ones — a publicly versioned API where consumers
need migration time. Internal-only code never qualifies.

## 4. No premature abstractions.

Three similar lines is better than a premature interface. Don't design
for hypothetical second implementations until they exist.

But once a second implementation exists (e.g. Postgres + memory drivers
behind `service.Repository`), the abstraction is real and must be used
correctly — not bypassed by one of the implementations.

## 5. Comments are for the why, never the what.

Default to writing no comments. Add one only when:

- The constraint is hidden (a workaround for a specific upstream bug,
  a subtle invariant that isn't visible in the code).
- A reader would be surprised by the choice and the rationale matters.

Never write:

- Comments restating what the code does.
- Comments referencing the current PR / fix / ticket.
- Comments listing the call sites of a function (they rot).
- Comments describing a previous version of the code.

## 6. No half-finished implementations.

If a feature is being added, it lands complete: the implementation, the
tests, the wiring, the documentation it warrants. Do not land a stub
with a TODO and ship the rest in a follow-up PR. If the work is too big
for one change, split it along feature boundaries (e.g. "Postgres
driver" + "Postgres CI integration" as separate complete units), not
along completeness boundaries ("the function" + "tests for it later").

## 7. Tests are part of the change, not after it.

Every new function gets tests. Every bug fix gets a regression test
that fails before the fix and passes after. Conformance suites for
swappable backends (e.g. `internal/repo/conformance/`) get extended,
not bypassed, when a new backend lands.

## 8. Commit messages: clean, human, no AI attribution.

- Imperative mood: "fix proto descriptor panic", not "fixed" or "fixing".
- The body explains the why when the diff doesn't.
- No "Generated by", "Co-Authored-By: Claude", "🤖", or any model
  attribution anywhere in the message or trailer.
- No reference to PR numbers from a previous repo, ticket IDs the
  reader has no access to, or "as discussed" hand-waves.

## 9. When you find existing violations, fix them.

You don't have to fix everything in one change, but if your change
touches an area with violations of these rules, fix the violations in
that area as part of your change. Do not "preserve the existing
pattern" when the existing pattern is wrong.

## 10. Pin every external version exactly. No floating tags.

Container images, language toolchains, lockfiles, every dependency that
gets pulled at build/run time pins to a specific patch version (or
better, a content digest):

- `postgres:16.13-alpine3.23` — never `:16-alpine`, never `:alpine`
- `GO_VERSION: '1.25.9'` — never `'1.25'`, never `'stable'`
- `go-version: '1.25.9'` in `actions/setup-go` — never `'1.25.x'`
- `golangci-lint@v1.62.2` — never `@latest`
- `protoc-gen-go v1.36.11` — pinned in `go install …@v1.36.11`

Why: builds must be reproducible. A floating tag means yesterday's
green CI doesn't tell you anything about today's image. A bug in a
patch release shows up as a "flaky test." A breaking change in a
silently-rolled image shows up as a production incident.

GitHub Actions are the one accepted exception: `actions/checkout@v4`,
`actions/setup-go@v5` etc. follow the convention that a major-version
tag is a stable line. Use major tags for actions; pin to a SHA only
when you need stronger supply-chain guarantees.

When upstream releases a new patch, **bump the pin in a deliberate
commit** with a clear changelog reference. Renovate / Dependabot can
open the PR; a human reviews and merges.

## 11. Run the PR review gate on every PR.

Every PR is reviewed by a multi-agent gate in three phases — **Triage →
Review → Verify** — before merge.

**Triage** classifies the diff (does it touch the contract/data surface;
does it touch UI), failing open if it can't tell.

**Review** runs reviewers in parallel, each owning ONE distinct
responsibility (no overlap). Five are always on:

1. **Product** — user/business value, scope, UX gaps, missing
   empty/error/loading states, shippability.
2. **Security** — authn/authz, secrets, injection/XSS/SSRF,
   open-redirect, signature/replay, data exposure, supply-chain.
3. **Performance** — hot-path cost, N+1/redundant I/O, blocking work,
   allocations, payload size, query/index efficiency, behaviour at load.
4. **Maintainability** — the rules in this file: clarity/naming, no
   shims (§1), dead code deleted (§2), right altitude (§3/§4), tests
   that prove new logic (§6/§7), idiomatic fit.
5. **Correctness** — pure bug-hunting: off-by-one, nil deref, inverted
   conditionals, races, leaks, overflow, edge cases, broken invariants.

Two more are added only when the diff warrants them:

- **Contract & Migration** — when proto/API/`gen/**`/entdb-schema or a
  DB migration changes: wire compatibility (no renumbered/reused proto
  tags or entdb field_ids/type_ids), generated-code drift, RPC signature
  breaks, migration forward+backward safety.
- **Accessibility & UX** — when UI files (`docs-site/**`, `*.astro`,
  etc.) change: keyboard/focus/ARIA, contrast, hit targets,
  screen-reader text, reduced-motion, complete states, i18n/RTL.

**Verify** hands every finding a reviewer marked *blocking* to a fresh,
skeptical agent prompted to refute it ("real and merge-blocking, or a
false positive / already handled?"). Only survivors become confirmed
blockers — this prevents blocking a merge on a plausible-but-wrong
finding.

**Gate decision:** APPROVED iff zero confirmed blockers survive
verification and no selected reviewer is missing — a reviewer's raw
REQUEST_CHANGES whose blocking findings were all refuted does not block.
Otherwise BLOCKED.

The gate runs as an agent workflow (`.claude/workflows/review-gate.js`)
**inside the Claude Code agent harness** — it depends on harness
primitives (`agent()`, `parallel()`), so it is not a plain `node` script
and external contributors cannot run it directly. It is a **maintainer
step**: the maintainer handling a PR (including PRs from outside
contributors) runs it and is accountable for resolving every confirmed
blocker before merge.

Run it on every PR — `Workflow({name: 'review-gate', args: <pr-number>})`.

The gate is **advisory**: it posts a `--comment` review and never
auto-approves or blocks the merge. It must never be granted
approve/merge authority. On this repo — a single-maintainer project
where the sole code owner cannot approve their own PRs — the required
**merge gate is the CI status checks (§10) plus a clean review-gate run**
with its confirmed blockers resolved; the maintainer merges on that
basis. When a second maintainer joins, restore a required human approval
on top. The gate runs alongside CI, never instead of it. It complements,
and does not replace, an ad-hoc `/code-review`-style pass.

## 12. When in doubt, prefer fewer lines, fewer files, fewer abstractions.

The job is not to add code — it is to express the system clearly. If
two paths are equally correct, pick the one that deletes more.
